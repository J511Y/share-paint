# share-paint Service Development Team

현재 `share-paint` 레포를 기준으로 에이전트팀 운영을 바로 시작할 수 있게 정리한 운영안입니다.

## 1) 팀 편성 (개발 에이전트)

- **PM Agent (`pm`)**
  - 역할: 요구사항 정제, 우선순위, 일정/리스크 관리
  - 주 책임: 이번 주 스프린트 목표 정의, 범위 변경 판단

- **Backend Agent (`backend`)**
  - 역할: API, DB, 인증/권한, WebSocket, 데이터 정합성
  - 주 책임: 백엔드 설계/구현 태스크 분해

- **Frontend Agent (`frontend`)**
  - 역할: 화면/상태관리/UX 설계 및 구현
  - 주 책임: MVS 화면 완성도 점검, 핵심 UX 플로우

- **Review Agent (`reviewer`)**
  - 역할: 품질검수, 누락점/보안/회귀점 검토
  - 주 책임: PR 전 점검 리스트 운영, DoD 충족 확인

- **DevOps Agent (`devops`)** *(선택)*
  - 역할: 빌드/배포/환경 변수/운영 안정성
  - 주 책임: 실행 체크리스트와 롤백/이슈 대응 정리

## 2) 이번 주 운영 규칙

- 대상 브랜치: `develop` → `feature/*`
- 작업 단위:
  - 1개 기능당 1개 이슈 기준
  - 완료 후 `feat|fix|refactor|chore` prefix 커밋
- 의사결정 원칙:
  - **Linear 우선순위 + PM 승인 순**
  - `MVP` 완료 기준 먼저 달성 후 확장 기능 진행

## 3) 공유 템플릿(에이전트에게 전달할 공통 포맷)

**목표**
- 1줄로 구현 목표 정의

**산출물**
- 파일/컴포넌트/쿼리 변경점
- 테스트/검증 결과

**제약**
- `branch: personal`
- `no_push_remote: true`
- `.env`는 하드코딩 금지

**Acceptance**
- 동작 조건(체크리스트)
- 누락/리스크 항목
- 다음 단계 제안

## 4) 지금 바로 시작할 추천 태스크

1. `AGENTS.md` 기준으로 **현재 상태 점검** (진행 체크)
2. `Linear` 또는 이슈 트래커 없는 경우엔 **핵심 3개 MVP** 우선순위 수립
3. 공통 스펙 정리(랜덤 주제 생성, 저장, 피드, 대결방 안정성) 기반으로 1차 스프린트 분할
